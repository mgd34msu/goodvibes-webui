# Decision: run the Playwright e2e suite in CI, non-blocking to start

Date: 2026-07-07

## Context

`goodvibes-webui`'s hermetic Playwright suite (`e2e/*.e2e.ts` — phone 390x844 and
desktop projects, both running against a mock daemon with no real network) existed
on disk but was never wired into `.github/workflows/ci.yml`. The typecheck/test/build
job and the non-blocking lint job ran on every push and pull request; the e2e suite
ran only when someone remembered to run it locally. With the desktop web UI effort
adding many new views over the next several briefs, the suite needed to run in CI or
its specs would rot unnoticed.

## Ruling

Add the suite as a separate job (`e2e`) in the existing CI workflow, initially
**non-blocking** (`continue-on-error: true` at the job level, the same pattern
already used by the `lint` job in this file), matching the prior W6-REL
recommendation. This is not a default choice — the ruling was to start non-blocking
specifically because the suite is new to CI infrastructure (as opposed to new to the
codebase) and its very first runs need room to shake out CI-only flake (browser
install path, cold-cache timing, viewport/animation timing differences between a
GitHub Actions runner and a local machine) without blocking every contributor's
merge on day one.

**Promotion criterion**: once the `e2e` job has produced 10 consecutive green runs
on `main` (scheduled or post-merge, not counting reruns of a flaked attempt) with
zero flake-driven reruns, remove `continue-on-error: true` from the job and drop the
promotion-criterion comment block above it in `ci.yml`. That is a one-line follow-up
change, not a redesign.

## What the job does

- Runs on `ubuntu-latest`, `timeout-minutes: 15` (suite itself is ~45s locally
  against a warm browser cache; the budget covers cold Chromium download + system
  dependency install on a fresh runner).
- Installs dependencies via the same pinned Bun toolchain as the other jobs
  (`bun-version: 1.3.14`).
- Caches the Playwright Chromium binary under `~/.cache/ms-playwright`, keyed on
  `${{ runner.os }}-playwright-<installed @playwright/test version>` (read from
  `node_modules/@playwright/test/package.json` after `bun install`, so a
  `package.json`/lockfile bump that changes the Playwright version automatically
  busts the cache and re-downloads the matching browser build). On a cache hit, the
  job still runs `playwright install-deps chromium` — the binary cache does not
  cover the runner's OS-level shared libraries, only the browser binary itself.
- Runs `bun run e2e` (`playwright test`, both the `phone` and `desktop` projects,
  against the suite's own hermetic mock-daemon vite dev server on port 4318 — never
  the real daemon, never ports 3421/4444).
- On failure, uploads `e2e/.artifacts/report` (the HTML reporter output) and
  `e2e/.artifacts/test-output` (traces retained on failure, screenshots on failure)
  as a build artifact (`playwright-report`, 14-day retention), so a red run is
  debuggable from the Actions UI without needing to reproduce locally first.

## Verification

- `actionlint` (run via the `rhysd/actionlint` Docker image, no local actionlint
  binary available) reported zero findings against the edited `ci.yml`.
- The from-nothing install path was proven directly: with `PLAYWRIGHT_BROWSERS_PATH`
  pointed at a freshly created empty directory (not the machine's existing
  `~/.cache/ms-playwright`), `bunx playwright install chromium` downloaded and
  installed Chromium + Chrome Headless Shell + ffmpeg from scratch, then
  `bun run e2e` against that same fresh path ran the full suite: 41 passed, 23
  skipped (project-scoped `only()` filtering — phone-only specs skip on the desktop
  project and vice versa), 0 failed, in 41.6s. This is the exact command sequence
  the CI job's cache-miss branch runs.
- Scope confirmed: this job is added to `goodvibes-webui/.github/workflows/ci.yml`
  only. It does not touch the SDK or agent repos' release trains.

## Why this shape, not something else

- **Not blocking from day one**: a suite that has never run in CI has an unknown
  CI-specific flake rate; making it blocking immediately risks a same-day revert
  driven by CI-only noise rather than real regressions, and the harness rots faster
  if it gets disabled reflexively than if it stays advisory a little longer.
- **Not skipped/deferred entirely**: a suite that runs nowhere is worse than a
  suite that runs non-blocking — the phone/desktop viewport proofs and the honesty
  states they check (delete-means-delete, degraded stream states, touch targets)
  are exactly the kind of regression a passing typecheck/build does not catch, and
  new views land in the very next briefs on this effort.
- **Same pattern as the existing `lint` job**, not a bespoke mechanism — the repo
  already had a precedent (job-level `continue-on-error: true`) for "runs on every
  push, visible in the Actions UI, does not block merge" before this change; reusing
  it keeps the workflow file consistent rather than introducing a second way to mark
  a job advisory.
